# Contributing

This repository is private and proprietary (see [LICENSE](LICENSE)).
This document exists so anyone given commit access — designer, hired
dev, future me — can ship a change without rediscovering the build
flow each time.

## Setup once per machine

```sh
git clone https://github.com/mrdst13/logbook.git
cd logbook
# No npm install needed for the main PWA build — build.js has zero deps.
# Only the Cloudflare Worker has its own package.json (under worker/logbook-api/).
```

## Working on the PWA (`src/`)

1. Create a branch off `main`:
   ```sh
   git checkout -b feat/your-short-description
   ```
2. Edit files under `src/` (never edit `logbook.html` directly — it is
   regenerated by the build).
3. Rebuild the bundle:
   ```sh
   node build.js
   ```
   This writes the new `logbook.html`, stamps `BUILD_VERSION` with
   `YYYY-MM-DD-<git-sha>`, and appends `?v=<version>` to every
   `logbook.html` link in `index.html`, `demo.html`, `security.html`,
   and `privacy.html` so browsers and the CDN pick up the fresh
   bundle.
4. Test locally — open `logbook.html` in a browser (Chrome / Safari).
   `index.html` is the landing page; click "Open my logbook →" to
   reach the app.
5. Commit your source changes **AND the rebuilt `logbook.html` +
   touched landing files**:
   ```sh
   git add src/ logbook.html index.html demo.html security.html privacy.html
   git commit -m "fix(area): short imperative describing the change"
   ```
   The build-check GitHub Action will reject any PR that changes
   `src/` without committing the rebuilt `logbook.html`.
6. Open a PR against `main`. Cloudflare Pages will publish the change
   to `flycumulo.ca` shortly after merge.

### Branch naming

| Prefix | Use for |
|---|---|
| `feat/` | New feature or visible UI change |
| `fix/` | Bug fix or correctness change |
| `polish/` | Visual / typographic / spacing tweak |
| `docs/` | README, CONTRIBUTING, comments only |
| `infra/` | Build, CI, deploy config |

Short kebab-case suffix. Example: `fix/recap-night-percentage`.

### Commit message style

Imperative subject line under 70 chars. Body explains why, not what,
ideally with the audit ID or design ref that motivated the change.
Single sentence is fine for trivial commits; multi-paragraph for
anything touching certifiable data, security, or vendor claims.

```
fix(recap): drop night-percentage from fun facts row

Audit copy editor flagged the "X% of your year flown in the dark" line
as morbid for an audience that flies night ops as routine work. Replaced
with a flight-count + average-length card that is equally factual but
identity-affirming.
```

## Working on the Worker (`worker/logbook-api/`)

The Cloudflare Worker that fronts Anthropic and Navblue lives in its
own subdirectory with its own README, `wrangler.jsonc`, and `package.json`.
Quick start:

```sh
cd worker/logbook-api
npm install          # one-time
npx wrangler dev     # local dev — http://localhost:8787
npx wrangler deploy  # ship to production
```

You must be authenticated against Cloudflare:

```sh
npx wrangler login
# opens a browser → Allow → token stored in ~/.config/.wrangler/
```

Secrets (e.g. `ANTHROPIC_API_KEY`) are managed via the Cloudflare
dashboard or `npx wrangler secret put NAME`. They are never committed
to the repo.

After a Worker deploy, verify with the curl tests in
`worker/README.md`.

## Hard rules

These rules protect users and the founder's reputation. They are
non-negotiable in every PR.

- **Never fabricate.** No invented testimonials, vendor commitments,
  certification claims, or regulatory interpretations. If a claim
  isn't directly verifiable, soften the wording or remove it.
- **Recalc never overwrites pilot values.** Auto-calculation helpers
  for night, XC, etc. only fill empty fields. Existing pilot entries
  are never silently rewritten.
- **No approximation in certifiable data.** Schedule is not actual.
  Block-off is not block-on. Empty is better than approximated.
- **Security-sensitive endpoints need spend caps.** Anything that
  proxies a paid API must ship with a documented per-request cap and
  a way to verify the upstream account has a daily spend limit.

## What's not in this repo (yet)

- Tests. There is no test suite — verification is manual today.
  Future test work belongs under `worker/logbook-api/test/` for the
  Worker and `tests/` at the repo root for the PWA.
- A CHANGELOG. Releases are tracked via git tags + the build version
  stamp at the bottom-right of every page.
